## Preparing for Production: Ensuring Reliability

**Think of this as your final quality check before going live.** These prerequisites ensure that when your workflow becomes a production API, it performs consistently and reliably for every user.

<Steps>
  <Step title="Successful Execution">
    **Run your workflow at least once successfully** in the ComfyUI workspace
    <br /> <br />
    *Why this matters:* This proves your workflow can complete its full process without errors, giving you and your users confidence in its reliability.
  </Step>
  <Step title="Save Workflow">
    **Save the workflow** after successful execution
    <br /> <br />
    *Why this matters:* This creates a stable version that can be consistently reproduced, preventing "works on my machine" issues in production.
  </Step>
  <Step title="Verify Dependencies">
    **Confirm all models and custom nodes** are properly set up
    <br /> <br />
    *Why this matters:* Dependencies that work in development need to be available in production. This step prevents runtime failures when users start making API calls.
  </Step>
</Steps>

## Defining Your API Interface: From Internal Logic to External Service

**Here's where your creative workflow becomes a professional API.** Right now, your workflow is perfectly designed for ComfyUI‚Äîbut APIs need clear, standardized interfaces that developers and applications can rely on.

This configuration step transforms your internal node connections into **a clean, documented API contract** that:

- **Creates intuitive parameters** that make sense to your users (not just ComfyUI experts)
- **Generates automatic documentation** that developers can understand and integrate with
- **Builds interactive interfaces** that non-technical users can navigate confidently
- **Establishes reliable contracts** that won't break when you update your internal workflow logic

<Note>
By default, workflows are not configured for deployment. You'll notice a **red cross mark** next to unconfigured workflows in your project's workflow sidebar.
</Note>

### The Interface Design Challenge

**Your workflow is complex and powerful‚Äîyour API interface should be simple and intuitive.** This is the art of good API design: hiding complexity while preserving capability.

FlowScale AI needs to understand:
- **Which inputs should be user-controllable** vs. which should remain internal constants
- **How outputs should be formatted** for different types of consumers (web apps, mobile apps, automation systems)
- **What level of control to expose** without overwhelming users or compromising your creative vision

## Using FlowScale I/O Nodes: The Bridge to Production APIs

**This is where ComfyUI nodes become API endpoints.** FlowScale I/O nodes are specially designed to translate between your creative workflow logic and the standardized interfaces that production applications expect.

<Tip>
**Design philosophy:** Instead of exposing every ComfyUI parameter as an API input, thoughtfully choose which controls your users actually need. Great APIs are simple to use but powerful in capability.
</Tip>

### Adding Input Nodes

<Steps>
  <Step title="Open Node Search">
    Double-click on an empty area in the ComfyUI workspace to open the node search
  </Step>
  <Step title="Search for Input Nodes">
    Type one of the following input node names:
    - **InputText** - For text prompts and string inputs
    - **InputImage** - For image uploads
    - **InputNumber** - For numeric parameters
    - **InputSlider** - For slider controls
  </Step>
  <Step title="Select FlowScale Nodes">
    Choose nodes with the **"[FS]"** prefix, indicating they are FlowScale-specific nodes
  </Step>
  <Step title="Connect to Workflow">
    Connect these input nodes to your workflow where external inputs are needed
  </Step>
</Steps>

### Your Workflow, Now Available Everywhere

**Here's the powerful part:** Your ComfyUI workflow can now be called from any application, website, mobile app, or automation system. The creative work you did locally is now global infrastructure.

<Note>
**What this means for your reach:** Instead of sharing screen recordings or files, you can now give developers a simple API call that brings your AI capabilities directly into their applications. Your creative work becomes a service that can scale to millions of users.
</Note>

<CodeGroup>

```bash curl
curl -X POST https://api.flowscale.ai/v1/run \
  -H "Authorization: Bearer your-api-key" \
  -H "Content-Type: application/json" \
  -d '{
    "workflow_id": "wf_abc123",
    "inputs": {
      "prompt": "A beautiful sunset over mountains",
      "steps": 20,
      "cfg": 7.5
    }
  }'
```

```javascript JavaScript
const response = await fetch('https://api.flowscale.ai/v1/run', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer your-api-key',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    workflow_id: 'wf_abc123',
    inputs: {
      prompt: 'A beautiful sunset over mountains',
      steps: 20,
      cfg: 7.5
    }
  })
});
```

```python Python
import requests

response = requests.post(
    'https://api.flowscale.ai/v1/run',
    headers={
        'Authorization': 'Bearer your-api-key',
        'Content-Type': 'application/json'
    },
    json={
        'workflow_id': 'wf_abc123',
        'inputs': {
            'prompt': 'A beautiful sunset over mountains',
            'steps': 20,
            'cfg': 7.5
        }
    }
)
```

</CodeGroup>

## Production Considerations

### Security & Authentication

- **API keys**: Secure access with regeneratable keys
- **Rate limiting**: Protect against abuse and manage costs
- **CORS configuration**: Control cross-origin access
- **Webhook support**: Real-time notifications for long-running jobs

### Performance Optimization

<AccordionGroup>
  <Accordion title="Caching Strategies">
    - Model caching to reduce cold start times
    - Result caching for identical requests
    - CDN integration for faster content delivery
  </Accordion>
  <Accordion title="Request Optimization">
    - Async processing for long-running workflows
    - Batch requests to reduce overhead
    - Progressive loading for large outputs
  </Accordion>
  <Accordion title="Resource Management">
    - Auto-scaling based on demand
    - GPU pooling for efficient utilization
    - Memory optimization for large models
  </Accordion>
</AccordionGroup>
